# 会议录制工具 - 技术探索记录

记录从零到最终方案的完整探索过程，包括每种方案的原理、尝试过程、遇到的问题和结论。

## 目标

在 macOS 上实现会议音频录制：
- 支持 Zoom / Teams / 飞书 / 腾讯会议
- 用户能正常听到会议声音
- 同时通过 BlackHole 虚拟声卡将音频录制到文件
- 一条命令启动/停止

## 方案一：多输出聚合设备（Stacked Aggregate Device）

### 原理

macOS CoreAudio 支持创建聚合设备（Aggregate Device），其中 `"stck": 1` 表示多输出（stacked）模式。音频数据会同时发送到所有成员设备：

```
App → 系统输出 → Meeting Multi-Output (聚合设备)
                    ├── 耳机/扬声器  (用户听到)
                    └── BlackHole 2ch (ffmpeg 录制)
```

通过 `AudioHardwareCreateAggregateDevice` API 动态创建，指定固定 UID `com.meeting-record.multi-output` 保证 Teams 等 App 能跨重建持久识别设备。

### 创建参数

```swift
let desc: [String: Any] = [
    "name": "Meeting Multi-Output",
    "uid": "com.meeting-record.multi-output",
    "private": 0,
    "stck": 1,          // 多输出模式
    "drift": 1,          // 启用 drift correction
    "subdevices": [
        ["uid": masterUID, "drift": 0],       // master 设备
        ["uid": secondaryUID, "drift": 1]     // secondary，带 drift correction
    ],
    "master": masterUID  // 指定时钟主设备
]
```

### 尝试 1：AirPods 做 master + BlackHole 做 secondary

**配置**：AirPods (24kHz) 做 master，BlackHole (48kHz) 做 secondary（带 drift correction）

**结果**：
- AirPods：✅ 有声音（用户能听到会议）
- BlackHole 录音：❌ 静音（-91 dB，完全无音频数据）

**原因**：master 设备的时钟频率为 24kHz，secondary 的 BlackHole 需要从 24kHz 上采样到 48kHz，但 BlackHole 驱动无法正确处理这种上采样，收到的全是零数据。

### 尝试 2：BlackHole 做 master + AirPods 做 secondary

**配置**：BlackHole (48kHz) 做 master，AirPods (24kHz) 做 secondary（带 drift correction）

**结果**：
- BlackHole 录音：✅ 正常（-24 dB ~ -30 dB，有效音频）
- AirPods：❌ 无声音

**原因**：master 时钟 48kHz，AirPods 作为 secondary 需要从 48kHz 降采样到 24kHz。drift correction 机制无法正确完成这个转换，AirPods 端收到的数据无效。

**测试结果**：尝试将 AirPods 改为 master（方案1）→ AirPods 无声，录音同样静音。两种方案均不可行。

### 结论

**AirPods (24kHz) + BlackHole (48kHz) 在聚合设备中完全不兼容。** 无论哪一方做 master，另一方都无法正常工作。核心问题是两个设备采样率不同（24kHz vs 48kHz），macOS 的 drift correction 机制无法处理这种差异。

### 尝试 3：改用 MacBook Speaker + BlackHole

放弃 AirPods，改用 MacBook Pro 内置扬声器（48kHz）。与 BlackHole（48kHz）采样率一致。

**配置**：BlackHole (48kHz) 做 master，MacBook Speaker (48kHz) 做 secondary

**结果**：
- BlackHole 录音：✅ 正常（-20.9 dB ~ -30.3 dB）
- MacBook Speaker：❌ 无声音

**意外！** 即使采样率一致，BlackHole 做 master 时 Speaker 作为 secondary 仍然没有声音。

### 尝试 4：MacBook Speaker 做 master + BlackHole 做 secondary

**结果**：
- MacBook Speaker：✅ 有声音
- BlackHole 录音：❌ 静音（-91 dB）

**结论**：与 AirPods 的情况完全一样——master 端正常，secondary 端静音。

### 尝试 5：去掉 drift correction 和 master 指定（no-drift 模式）

猜想可能是 drift correction 或 master 指定导致的问题，尝试两个设备平等对待：

```swift
// 无 master，无 drift correction
let desc: [String: Any] = [
    "name": kMOName, "uid": kMOUID,
    "private": 0, "stck": 1,
    "subdevices": [
        ["uid": "BuiltInSpeakerDevice"],
        ["uid": "BlackHole2ch_UID"]
    ]
]
```

**Speaker 放第一个位置**：
- Speaker：✅ 有声音
- BlackHole 录音：❌ 静音

**BlackHole 放第一个位置**：
- Speaker：❌ 无声音
- BlackHole 录音：未测试（Speaker 已经没声了）

**结论**：无论有无 drift correction，无论是否指定 master，stacked aggregate 设备中只有第一个子设备（或 master 设备）能正常输出音频，第二个设备始终静音。

### 尝试 6：系统自带的 Multi-Output Device

系统中已存在一个通过 Audio MIDI Setup 创建的 `Multi-Output Device (Speakers + BlackHole)`（ID: 122）。

**结果**：❌ 也没有声音

**结论**：不是我们代码创建方式的问题，macOS 多输出设备在当前环境下就是无法同时让两个设备都正常工作。

### 方案一最终结论

**macOS 多输出聚合设备不可靠。** 在我们的测试环境中，始终只有一个子设备能正常输出音频。这可能与以下因素有关：
- BlackHole 作为虚拟设备的特殊性
- macOS 版本对聚合设备的支持差异
- 音频会话管理的内部逻辑

## 方案二：AVAudioEngine 实时转发

### 原理

放弃聚合设备，改为：
1. 系统输出直接设为 BlackHole（ffmpeg 从此录音）
2. 用 AVAudioEngine 从 BlackHole 读取音频，实时转发到耳机/扬声器
3. AVAudioEngine 号称能自动处理采样率转换

```
App → 系统输出 → BlackHole 2ch
                    ├── ffmpeg 录制
                    └── AVAudioEngine → 耳机/扬声器（实时转发）
```

### 实现

```swift
let engine = AVAudioEngine()

// 设置输入设备为 BlackHole
let inputAU = engine.inputNode.audioUnit!
AudioUnitSetProperty(inputAU, kAudioOutputUnitProperty_CurrentDevice, ...)

// 设置输出设备为扬声器
let outputAU = engine.outputNode.audioUnit!
AudioUnitSetProperty(outputAU, kAudioOutputUnitProperty_CurrentDevice, ...)

// 连接
let inputFormat = engine.inputNode.outputFormat(forBus: 0)
engine.connect(engine.inputNode, to: engine.mainMixerNode, format: inputFormat)

try engine.start()
```

### 尝试 1：BlackHole → AirPods

**结果**：❌ 启动失败

```
Error Domain=com.apple.coreaudio.avfaudio Code=-10875 "(null)"
UserInfo={false condition=IsFormatSampleRateAndChannelCountValid(outputHWFormat)}
```

**原因**：AVAudioEngine 内部会获取输出设备的硬件格式（hardware format），AirPods 的 24kHz 不在 AVAudioEngine 认为有效的格式范围内，直接拒绝。

### 尝试 2：BlackHole → MacBook Speaker

两个设备都是 48kHz，理论上不应该有格式问题。

**结果**：❌ 同样启动失败，报完全相同的错误

```
Error Domain=com.apple.coreaudio.avfaudio Code=-10875 "(null)"
UserInfo={false condition=IsFormatSampleRateAndChannelCountValid(outputHWFormat)}
```

**原因**：AVAudioEngine 在使用自定义设备（非系统默认设备）时，获取 hardware format 的方式有问题。当输出设备不是系统当前默认输出时（此时系统默认输出已被切到 BlackHole），AVAudioEngine 可能获取到错误的格式信息。

### 方案二结论

**AVAudioEngine 不适合用于自定义设备间的音频转发。** 它的设备管理过于高层抽象，在非标准设备组合下无法正确初始化。

## 方案三：底层 AudioUnit (AUHAL) 实时转发 ✅ 最终方案

### 原理

使用底层的 Audio Unit HAL Output (`kAudioUnitSubType_HALOutput`) API，完全手动控制输入和输出设备、数据格式、缓冲区。不依赖任何高层封装。

架构：

```
App → 系统输出 → BlackHole 2ch ─┬→ ffmpeg 录制到文件
                                 └→ audio-monitor (AUHAL) → 扬声器/耳机
```

### 实现要点

#### 1. 创建两个独立的 AUHAL AudioUnit

```swift
// 输入 AU：从 BlackHole 采集音频
var inputUnit: AudioUnit?
AudioComponentInstanceNew(comp, &inputUnit)
// 启用输入 bus (element 1)，禁用输出 bus (element 0)
AudioUnitSetProperty(inputAU, kAudioOutputUnitProperty_EnableIO, kAudioUnitScope_Input, 1, &enableIO, ...)
AudioUnitSetProperty(inputAU, kAudioOutputUnitProperty_EnableIO, kAudioUnitScope_Output, 0, &disableIO, ...)
// 指定输入设备
AudioUnitSetProperty(inputAU, kAudioOutputUnitProperty_CurrentDevice, kAudioUnitScope_Global, 0, &inputDevID, ...)

// 输出 AU：播放到扬声器
var outputUnit: AudioUnit?
AudioComponentInstanceNew(comp, &outputUnit)
// 默认就是 output enabled, input disabled
AudioUnitSetProperty(outputAU, kAudioOutputUnitProperty_CurrentDevice, kAudioUnitScope_Global, 0, &outputDevID, ...)
```

#### 2. 统一中间格式

两个 AU 之间使用 Float32 PCM 格式通信，采样率跟随输入设备：

```swift
var streamFmt = AudioStreamBasicDescription(
    mSampleRate: inputFmt.mSampleRate,  // 48000
    mFormatID: kAudioFormatLinearPCM,
    mFormatFlags: kAudioFormatFlagIsFloat | kAudioFormatFlagIsPacked | kAudioFormatFlagIsNonInterleaved,
    mBytesPerPacket: 4,
    mFramesPerPacket: 1,
    mBytesPerFrame: 4,
    mChannelsPerFrame: inputFmt.mChannelsPerFrame,  // 2
    mBitsPerChannel: 32,
    mReserved: 0)
```

#### 3. 环形缓冲区连接输入和输出

输入回调（在音频线程上被调用）从 BlackHole 读取数据写入环形缓冲区：

```swift
let inputCallback: AURenderCallback = { (inRefCon, ioActionFlags, inTimeStamp, inBusNumber, inNumberFrames, ioData) -> OSStatus in
    // AudioUnitRender 从输入设备读取 PCM 数据
    AudioUnitRender(ctx.inputAU, ioActionFlags, inTimeStamp, 1, inNumberFrames, bufList)
    // 写入环形缓冲区
    // ...
    return noErr
}
```

输出回调（在另一个音频线程上被调用）从环形缓冲区读取数据送给扬声器：

```swift
let outputCallback: AURenderCallback = { (inRefCon, ioActionFlags, inTimeStamp, inBusNumber, inNumberFrames, ioData) -> OSStatus in
    // 从环形缓冲区读取数据
    // 如果缓冲区没有足够数据（欠载），输出静音
    // ...
    return noErr
}
```

#### 4. 关键的 Bus 编号

AUHAL 的 bus 编号容易搞混：
- **Bus 0 (Element 0)**：输出端（Output scope = 送给硬件，Input scope = 你提供的数据）
- **Bus 1 (Element 1)**：输入端（Input scope = 来自硬件，Output scope = 你读取的数据）

所以：
- 输入 AU 从 bus 1 读数据：`AudioUnitRender(inputAU, ..., bus: 1, ...)`
- 输入 AU 的输出格式设在 bus 1 的 Output scope：`kAudioUnitScope_Output, element: 1`
- 输出 AU 的输入格式设在 bus 0 的 Input scope：`kAudioUnitScope_Input, element: 0`

### 测试结果

```
Input:  BlackHole2ch_UID — 48000.0Hz, 2ch
Output: BuiltInSpeakerDevice — 48000.0Hz, 2ch
Monitoring: BlackHole2ch_UID → BuiltInSpeakerDevice
```

- MacBook Speaker：✅ 能听到声音
- BlackHole 录音（ffmpeg）：✅ 正常（-32.0 dB）

**两端同时正常工作！**

### 为什么 AUHAL 能成功而聚合设备不行？

1. **聚合设备**让 CoreAudio 内核层面尝试同时驱动两个硬件设备，受限于内核态的时钟同步和数据分发机制
2. **AUHAL** 在用户态独立读写两个设备，输入和输出各自运行在自己的音频线程上，通过应用层的环形缓冲区连接，完全绕开了内核态的设备同步问题

## 其他尝试和发现

### 尝试修改设备采样率

```bash
# 尝试将 AirPods 设为 48kHz
~/bin/audio-multi-output set-sample-rate 140 48000
# 返回成功，但实际查询仍然是 24000
```

**结论**：蓝牙设备的采样率由蓝牙编解码器决定，API 调用虽然返回成功但实际不生效。AirPods Pro 使用的 AAC 编解码器固定在 24kHz。

### CoreAudio setDefaultOutput 对聚合设备无效

```swift
// 这个调用对普通设备有效，但对 stacked aggregate 设备无效
AudioObjectSetPropertyData(kAudioObjectSystemObject,
    &addr, // kAudioHardwarePropertyDefaultOutputDevice
    0, nil, size, &deviceID)
```

必须使用 `SwitchAudioSource`（基于 `AudioDeviceSetPropertyData` 的旧 API）才能切换到聚合设备。这是 macOS 的一个已知行为。

### ffmpeg avfoundation 不支持 -ar / -sample_rate

```bash
# 这些都不工作
ffmpeg -f avfoundation -ar 48000 -i ":BlackHole 2ch" ...
ffmpeg -f avfoundation -sample_rate 48000 -i ":BlackHole 2ch" ...
```

avfoundation 输入格式不接受采样率参数，它使用设备的原生采样率。

### Zoom 配置文件

Zoom 的设备配置存储在 `~/Library/Application Support/zoom.us/data/viper.ini`，设备名用 UTF-8 的十六进制编码：

```ini
AECSPK=53616d652061732053797374656d    # "Same as System"
```

可以在 Zoom 未运行时通过 sed 直接修改。

### Teams 和腾讯会议

- Teams 配置存储在 WebView LevelDB 中，无法安全外部修改
- 腾讯会议使用自定义 GLS 二进制格式，同样无法外部修改
- 两者都需要用户手动在 App 内选择扬声器设备

## 最终架构

```
┌─────────────────────────────────────────────────┐
│                    macOS 系统                     │
│                                                   │
│  Zoom/Teams/飞书 → 系统输出 → BlackHole 2ch      │
│                                    │              │
│                          ┌─────────┼──────────┐  │
│                          │         │          │  │
│                          ▼         ▼          │  │
│                     ffmpeg录制  audio-monitor  │  │
│                     (录音文件)  (AUHAL转发)    │  │
│                                    │          │  │
│                                    ▼          │  │
│                              MacBook Speaker  │  │
│                              (用户听到声音)    │  │
│                          └────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 工作流程

**start**:
1. 记录当前输出设备（如 MacBook Pro Speakers）
2. 切换系统输出到 BlackHole 2ch
3. 启动 audio-monitor：BlackHole → 原始设备（实时转发）
4. 启动 ffmpeg：从 BlackHole 录制到 .m4a 文件

**stop**:
1. 停止 ffmpeg（SIGINT 优雅退出）
2. 停止 audio-monitor
3. 恢复系统输出到原始设备

### 对会议 App 的要求

| App | 扬声器设置 | 原因 |
|-----|-----------|------|
| Zoom | Same as System | 系统输出切到 BlackHole 后自动跟随 |
| Teams | BlackHole 2ch | 无 System Default 选项，直接选 BlackHole |
| 腾讯会议 | BlackHole 2ch | 同 Teams |
| 飞书 | 默认即可 | 跟随系统 |

## 关键教训

1. **macOS 多输出聚合设备不可靠**：即使采样率一致，也可能只有一个子设备正常工作。不要依赖它做关键音频路由。

2. **AVAudioEngine 不适合自定义设备场景**：它对设备格式校验过于严格，且获取非默认设备的硬件格式时行为不正确。

3. **底层 AudioUnit API 是最可靠的**：虽然代码量大、概念复杂（bus 编号、scope、callback），但给了完全的控制权。

4. **蓝牙设备采样率不可改**：由蓝牙编解码器决定，API 调用可能返回成功但实际不生效。

5. **环形缓冲区是连接异步音频管线的标准方式**：输入和输出在不同线程、不同节奏运行，环形缓冲区是最自然的解耦方式。

6. **每个会议 App 的音频设备管理方式都不同**：Zoom 用 INI 文件（可自动修改），Teams 用 LevelDB（不可修改），腾讯会议用二进制格式（不可修改），飞书默认跟随系统（无需修改）。

### 各组件职责

#### meeting-record.sh（主控脚本）

用户直接交互的入口，负责整个录制生命周期的编排：

| 命令 | 职责 |
|------|------|
| `start` | 记录当前设备 → 切到 BlackHole → 启动 audio-monitor → 启动 ffmpeg |
| `stop` | 停 ffmpeg → 停 audio-monitor → 恢复原始设备 |
| `status` | 查询录制状态 |
| `setup` | 一键配置 Zoom/Teams/腾讯会议扬声器 |
| `setup-zoom` | 自动修改 Zoom 配置文件（INI hex 编码） |
| `setup-teams` | 引导用户在 Teams 中手动选择 BlackHole 2ch |
| `setup-wemeet` | 引导用户在腾讯会议中手动选择 BlackHole 2ch |

**依赖**：audio-multi-output、audio-monitor、ffmpeg、SwitchAudioSource

#### audio-multi-output（CoreAudio 设备管理）

Swift 编写的命令行工具，封装 CoreAudio API，提供设备查询和管理：

| 功能 | 说明 |
|------|------|
| 设备查询 | 列出设备、查 UID/名称/ID、查默认设备、查 BlackHole |
| 设备控制 | 设置默认输出、设置/查询采样率 |
| 聚合设备管理 | 创建/销毁多输出设备、清理遗留设备 |

**编译**：`swiftc -O -framework CoreAudio -framework AudioToolbox audio-multi-output.swift -o audio-multi-output`

**注意**：聚合设备功能在最终方案中不再使用，但保留用于调试和未来实验。

#### audio-monitor（AUHAL 音频转发）

Swift 编写的实时音频转发工具，核心组件。使用底层 AudioUnit HAL Output API：

```
输入设备 (BlackHole) → 输入 AUHAL → 环形缓冲区 → 输出 AUHAL → 输出设备 (Speaker)
```

内部结构：
- **输入 AudioUnit**：配置为仅输入模式（enable input bus 1, disable output bus 0），绑定到 BlackHole
- **输出 AudioUnit**：默认播放模式，绑定到扬声器/耳机
- **环形缓冲区**：65536 帧 × 通道数，连接两个 AU 的音频线程
- **输入回调**：AudioUnitRender 从 BlackHole 读 PCM → 写入环形缓冲区
- **输出回调**：从环形缓冲区读 PCM → 填入输出缓冲区（欠载时输出静音）

**编译**：`swiftc -O -framework CoreAudio -framework AudioToolbox audio-monitor.swift -o audio-monitor`

#### 外部依赖

| 软件 | 作用 | 安装 |
|------|------|------|
| BlackHole 2ch | 虚拟音频回环设备，48kHz | `brew install blackhole-2ch` |
| ffmpeg | 从 BlackHole 采集音频并编码为 AAC | `brew install ffmpeg` |
| SwitchAudioSource | 切换系统默认音频设备 | `brew install switchaudio-osx` |

### 组件间调用关系

`meeting-record.sh` 是中心编排者：

```
meeting-record.sh start:
    ① audio-multi-output default-output-uid    → 获取当前输出设备 UID
    ② audio-multi-output default-output-name   → 获取当前输出设备名称
    ③ audio-multi-output blackhole-uid          → 获取 BlackHole UID
    ④ SwitchAudioSource -s "BlackHole 2ch"      → 切换系统输出
    ⑤ audio-monitor $BH_UID $ORIG_UID &         → 后台启动音频转发
    ⑥ ffmpeg -f avfoundation -i ":BlackHole" &  → 后台启动录音

meeting-record.sh stop:
    ① kill -INT $FFMPEG_PID                     → 停止录音
    ② kill $MONITOR_PID                         → 停止音频转发
    ③ SwitchAudioSource -s "$ORIG_NAME"         → 恢复原始输出
```

### 数据流

```
运行时数据流（录制状态）:

  会议App音频 ──→ macOS音频系统 ──→ BlackHole 2ch (系统默认输出)
                                        │
                    ┌───────────────────┤
                    │                   │
                    ▼                   ▼
              audio-monitor        ffmpeg进程
              (AUHAL转发)          (avfoundation采集)
                    │                   │
                    ▼                   ▼
           MacBook Speaker     ~/Record/meeting_*.m4a
           (用户听到声音)       (AAC 128kbps 单声道)


控制流（状态管理）:

  meeting-record.sh ──→ /tmp/meeting-record/
                            ├── ffmpeg.pid       (录音进程ID)
                            ├── monitor.pid      (转发进程ID)
                            ├── original-output.uid  (原始设备，用于恢复)
                            ├── original-output.name
                            └── output-file      (当前录音文件路径)
```
