# MeeTap - 开发说明

## 项目概述

通过 macOS CoreAudio API 动态管理聚合音频设备，配合 BlackHole 虚拟声卡和 ffmpeg，实现会议内部音频的无感录制。

## 核心架构

### 音频路由原理

```
正常状态:
  App → 系统输出 → MacBook Speaker / 有线耳机

录制状态:
  App → 系统输出 → Meeting Multi-Output (聚合设备)
                      ├── MacBook Speaker / 有线耳机  (用户听到)
                      └── BlackHole 2ch (ffmpeg 录制)
```

**重要约束**：聚合设备中的所有成员设备必须采样率一致（均为 48kHz）。
MacBook 内置扬声器（48kHz）和有线耳机（48kHz）与 BlackHole（48kHz）兼容。
蓝牙耳机如 AirPods（24kHz）因采样率不匹配无法正常工作。

### 关键设计决策

#### 1. 固定 UID 的聚合设备

```
UID:  com.meeting-record.multi-output
Name: Meeting Multi-Output
```

**为什么不用系统自带的多输出设备？**
- 系统多输出设备的成员在创建后不能通过 API 修改
- 换耳机后需要删除重建，但 Teams 会丢失对旧设备的引用

**为什么用固定 UID？**
- Teams 不支持 "System Default"，只能选择具体设备
- 固定 UID 保证每次重建后 Teams 自动重连到同一个设备
- Zoom 也可以直接选这个设备，不依赖 "Same as System"

#### 2. stop 时不销毁设备

```bash
# stop_recording() 中的关键注释:
# 注意：不销毁多输出设备！
# Teams 不支持 "System Default"，它记住的是具体设备名。
# 保留 Meeting Multi-Output 设备让 Teams 下次还能找到它。
```

设备保留在系统中不影响正常使用，因为系统输出已切回原始设备。
下次 `start` 时会先 cleanup 再重建（更新成员为当前耳机）。

#### 3. 当前输出是多输出设备时的处理

如果用户在 Meeting Multi-Output 已是系统默认输出时执行 `start`，
脚本会从状态文件中读取上次记录的真实耳机 UID，避免嵌套聚合设备。

## audio-multi-output 工具

### 编译

```bash
swiftc -O -framework CoreAudio -framework AudioToolbox \
  ~/bin/audio-multi-output.swift -o ~/bin/audio-multi-output
```

### 命令列表

| 命令 | 说明 | 输出 |
|------|------|------|
| `list` | 列出所有输出设备 | `name\|uid\|id` 每行一个 |
| `default-output-uid` | 当前默认输出设备 UID | UID 字符串 |
| `default-output-name` | 当前默认输出设备名称 | 设备名 |
| `default-output-id` | 当前默认输出设备 ID | 数字 ID |
| `default-input-name` | 当前默认输入设备名称 | 设备名 |
| `blackhole-uid` | BlackHole 设备 UID | UID 字符串 |
| `create <master-uid> <secondary-uid>` | 创建聚合设备 | 新设备 ID |
| `destroy <device-id>` | 销毁聚合设备 | 无 |
| `set-default <device-id>` | 设置默认输出 | 无 |
| `find-uid <uid>` | 通过 UID 查找设备 ID | 数字 ID |
| `cleanup` | 清理所有 Meeting Multi-Output 设备 | 无 |

### CoreAudio API 要点

- `AudioHardwareCreateAggregateDevice` 创建聚合设备
  - `"stck": 1` 表示 stacked (多输出) 模式，非 aggregate (聚合输入) 模式
  - `"master"` 指定时钟主设备
  - `"drift": 1` 启用 drift correction（采样率同步）
  - `"uid"` 可自定义，保证跨重建一致性
- `AudioHardwareDestroyAggregateDevice` 销毁聚合设备
- 聚合设备在进程退出后仍然存在，直到被显式销毁或系统重启
- **CoreAudio `setDefaultOutput` 对 stacked aggregate 设备无效**，必须使用 `SwitchAudioSource` 切换

### 关键坑：BlackHole 必须做 master

BlackHole 做 master 设备，耳机/扬声器做 secondary（带 drift correction）。
这样 ffmpeg 能从 BlackHole 端正常录到音频。

### 关键坑：蓝牙耳机（AirPods）采样率不兼容

**AirPods 等蓝牙耳机无法与 BlackHole 组成多输出设备正常工作。** 这是目前已知的核心限制。

| 配置 | 录音（BlackHole） | 耳机（AirPods） |
|------|-------------------|-----------------|
| AirPods 做 master (24kHz) | ❌ 静音 (-91 dB) | ✅ 有声 |
| BlackHole 做 master (48kHz) | ✅ 正常 (-30 dB) | ❌ 无声 |

**原因**：AirPods 蓝牙音频固定 24kHz，BlackHole 固定 48kHz，两者无法通过 `setSampleRate` 调整。macOS stacked aggregate 设备的 drift correction 无法正确处理这种采样率差异——无论哪一方做 master，另一方都会出问题。

**AVAudioEngine 方案也不可行**：尝试过用 AVAudioEngine 做实时转发（BlackHole → AirPods），但 AVAudioEngine 在输出设备 hardware format 为 24kHz 时直接报错 `Code=-10875 IsFormatSampleRateAndChannelCountValid(outputHWFormat)`。

**当前解决方案**：使用 MacBook Pro 内置扬声器（48kHz）或有线耳机（48kHz）代替 AirPods。两者与 BlackHole 采样率一致，多输出设备工作完全正常。

**潜在未来方案**：
- 使用底层 AudioUnit API（而非 AVAudioEngine）实现采样率转换转发
- 等待 BlackHole 驱动支持动态采样率
- 使用第三方虚拟声卡（如 Loopback）可能支持自适应采样率

## meetap 流程

### start 流程

```
检查是否已在录制 → 是 → 提示并退出
                   ↓ 否
检查依赖 (audio-multi-output, ffmpeg, BlackHole)
                   ↓
获取当前默认输出设备 (UID + Name)
                   ↓
当前输出是 Meeting Multi-Output? → 是 → 从状态文件读取真实耳机
                   ↓ 否
当前输出是 BlackHole? → 是 → 提示并退出
                   ↓ 否
cleanup 旧的多输出设备
                   ↓
create 新的多输出设备 (当前耳机 + BlackHole)
                   ↓
set-default 到新的多输出设备
                   ↓
启动 ffmpeg 后台录制 (从 BlackHole 2ch 采集)
                   ↓
验证 ffmpeg 进程存活 → 失败 → 回滚并退出
                   ↓ 成功
启动 autostop_daemon（音频静音检测守护进程）
                   ↓
保存状态文件，输出录制信息
```

### stop 流程

```
检查 PID 文件是否存在 → 否 → 提示并退出
                       ↓ 是
发送 SIGINT 给 ffmpeg (优雅停止，确保文件尾完整)
                       ↓
等待 ffmpeg 退出
                       ↓
停止 audio-monitor
                       ↓
从状态文件读取原始设备名
                       ↓
SwitchAudioSource 恢复 → 成功 → 提示已恢复
                        → 失败 → 提示手动恢复（设备可能已断开）
                       ↓
显示录音文件信息 (路径、大小、时长)
                       ↓
会话目录为空? → 是 → 删除空目录
              → 否 → 后台启动 transcribe_recording()
```

### transcribe 流程（后台自动执行）

```
检查 AWS CLI 和凭证
            ↓
创建 S3 桶 (meetap-transcribe-{account-id}，复用)
            ↓
上传录音文件到 S3
            ↓
启动 AWS Transcribe 作业 (en-US, 说话人分离, 最多10人)
            ↓
轮询等待完成 (每10秒)
            ↓
下载 JSON 结果 → transcribe-raw.json
            ↓
Python 解析: 按说话人分段 → transcript.txt
            ↓
清理 S3 文件和 Transcribe 作业
            ↓
macOS 通知: "转录完成"
            ↓
调用 summarize_transcript()
            ↓
boto3 converse_stream 调用 Claude Sonnet 4.6 (Bedrock)
流式接收会议纪要 → meeting-notes.md
            ↓
macOS 通知: "会议纪要已生成"
```

**设计要点**：
- 后台执行（`& disown`），不阻塞终端
- S3 桶名含 AWS Account ID，避免命名冲突，跨会话复用
- 转录完成后自动清理 S3 临时文件和 Transcribe 作业记录
- 会议纪要通过 boto3 `converse_stream` 流式生成，避免同步调用超时
- 使用独立 Python venv（`meetap-venv`），不污染系统 Python 环境
- 失败时发通知提醒，日志写入 `transcribe.log`

### autostop 流程（音频静音检测）

录制启动后自动运行的后台守护进程：

```
每 30 秒采样 BlackHole 音频电平（ffmpeg volumedetect，5 秒）
            ↓
mean_volume <= -50dB? → 否 → 重置静音计时器，继续循环
            ↓ 是
首次静音? → 记录 silence_start 时间
            ↓
已持续静音 >= 120 秒? → 否 → 继续循环
            ↓ 是
发送 macOS 通知
            ↓
执行 meetap stop（触发转录 + 纪要生成）
```

**设计要点**：
- 基于音频电平而非进程检测，不受会议 App 常驻后台影响
- -50dB 阈值在静音（约 -91dB）和有效音频之间留有足够余量
- 中间检测到任何声音即重置计时器，避免会议中短暂静默触发误停

### setup-zoom 流程

```
读取 ~/Library/Application Support/zoom.us/data/viper.ini
                       ↓
AECSPK 已是目标值? → 是 → 完成
                    ↓ 否
Zoom 正在运行? → 是 → AppleScript 退出 Zoom → 等待退出
                ↓
sed 修改 AECSPK 和 AECBSSPK (hex 编码的设备名)
                ↓
如果之前退出了 Zoom → 重新打开
```

Zoom 配置文件格式：`viper.ini`，INI 格式，设备名为 UTF-8 hex 编码。
- `AECSPK` — 扬声器
- `AECBSSPK` — 蓝牙通话扬声器
- `AECMIC` — 麦克风

### setup-teams 流程

Teams 新版 (native macOS) 的音频设置存储在 WebView LevelDB 中，
无法从外部安全修改。采用半自动方式：
1. 创建 Meeting Multi-Output 设备
2. 通过 `msteams://settings/devices` URL scheme 打开设置页
3. 引导用户手动选择

### setup-wemeet 流程

腾讯会议使用自定义 GLS 二进制格式 (.tk/.tv 键值对) 存储配置，
同样无法从外部安全修改。采用与 Teams 相同的半自动方式：
1. 确保 Meeting Multi-Output 设备存在（不存在则创建）
2. 打开腾讯会议应用
3. 引导用户进入「设置 → 音频 → 扬声器」手动选择

## Zoom 配置文件详解

路径：`~/Library/Application Support/zoom.us/data/viper.ini`

```ini
[APE]
AECSPK=<hex-encoded-device-name>    # 扬声器
AECBSSPK=<hex-encoded-device-name>  # 蓝牙通话扬声器
AECMIC=<hex-encoded-device-name>    # 麦克风
AECBSMIC=<hex-encoded-device-name>  # 蓝牙通话麦克风
```

编解码方式：
```bash
# 编码
echo -n "Meeting Multi-Output" | xxd -p | tr -d '\n'
# → 4d656574696e67204d756c74692d4f7574707574

# 解码
echo -n "4d656574696e67204d756c74692d4f7574707574" | xxd -r -p
# → Meeting Multi-Output
```

**注意**：Zoom 运行时会在退出时将 UI 状态写回文件，因此修改配置必须在 Zoom 未运行时进行。

## 腾讯会议配置文件详解

- **应用路径**：`/Applications/TencentMeeting.app`
- **Bundle ID**：`com.tencent.meeting`
- **数据容器**：`~/Library/Containers/com.tencent.meeting/`

### 配置存储格式

腾讯会议使用自定义 `GLS_BINARY_LSB_FIRST` 二进制格式：
- `.tk` 文件存储键名
- `.tv` 文件存储值

路径：`~/Library/Containers/com.tencent.meeting/Data/Library/Global/Preferences/`

### 已知的音频相关键

| 键名 | 说明 |
|------|------|
| `audio.device.speaker.last_auto_select` | 上次自动选择的扬声器 |
| `audio.device.mic.last_auto_select` | 上次自动选择的麦克风 |
| `audio.device.auto.select.enable` | 是否启用自动设备选择 |
| `audio.denoise.nslevel.local` | 本地降噪级别 |

### 为什么不能自动修改

- 二进制格式没有公开的序列化/反序列化规范
- 修改二进制文件可能导致校验失败或配置损坏
- 腾讯会议运行时可能覆盖外部修改

## 已知限制

| 限制 | 原因 | 潜在解决方案 |
|------|------|-------------|
| 外放场景下录音有回声 | 麦克风拾取扬声器播出的对方声音 | 戴耳机录制（推荐） |
| Teams/腾讯会议设置需手动点一下 | 自定义二进制/LevelDB 无法安全外部修改 | 尝试通过 Accessibility API 自动化 UI 操作 |
| 重启后多输出设备消失 | 聚合设备不跨重启持久化 | start 时自动重建（已处理），各 App 在设备重现后自动重连 |

> **已解决的历史限制**：
> - ~~蓝牙耳机（AirPods）不支持~~ → 已通过 AUHAL AudioUnit 实时转发解决采样率差异问题
> - ~~转录固定为 en-US~~ → 已改为自动语言检测（支持 en-US、zh-CN、ja-JP）
> - ~~只录系统音频，不录麦克风~~ → 已默认混录系统音频 + 麦克风（amix）
